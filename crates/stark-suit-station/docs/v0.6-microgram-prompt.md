## Mission — Session Goal

Promote `bms_frame_v1.md` recordings into the existing rsk-core microgram registry as `NVG-PWR-BMS-REPLAY-001`. The microgram doesn't run BMS hardware — it runs the **classification rules** (`classify_power_state` from `stark-suit-station/src/loops.rs`) against recorded frames and asserts the expected outcome. The trace is the test corpus; the microgram is the assertion shell.

### Thrust B — Define `NVG-PWR-BMS-REPLAY-001` shell

The microgram has:
- `id`: `NVG-PWR-BMS-REPLAY-001`
- `inputs`: `bms_frame` (object matching `BmsFrame` v1 schema)
- `tree`: classification rules pulled verbatim from `classify_power_state`:
  - `cell_temp_c >= 60.0 OR soc_pct <= 5.0` → `"Emergency"`
  - `cell_temp_c >= 45.0 OR soc_pct <= 15.0` → `"Critical"`
  - `cell_temp_c >= 35.0 OR soc_pct <= 30.0` → `"Caution"`
  - else → `"Nominal"`
- `outputs`: `power_state: string`
- `tests`: each test case is one frame from a recorded NDJSON trace + the expected `power_state`.

The decision tree is the rule set, not the recorded trace. The trace becomes the test corpus.

**Acceptance**: `mcg test NVG-PWR-BMS-REPLAY-001` runs all bundled test cases green using existing rsk-core tooling. No new runtime; reuse what the other 460+ micrograms do.

## Pseudocode Skeleton — Thrust B + C

```
ALGORITHM Microgram_Shell

    YAML NVG-PWR-BMS-REPLAY-001
        id: "NVG-PWR-BMS-REPLAY-001"
        version: 1
        title: "Power state classification from BMS frame"
        inputs:
          bms_frame:
            type: object
            schema_ref: "../../nexcore/crates/stark-suit-station/docs/bms_frame_v1.md"
        tree:
          - condition: "cell_temp_c >= 60.0 || soc_pct <= 5.0"
            output:    { power_state: "Emergency" }
          - condition: "cell_temp_c >= 45.0 || soc_pct <= 15.0"
            output:    { power_state: "Critical" }
          - condition: "cell_temp_c >= 35.0 || soc_pct <= 30.0"
            output:    { power_state: "Caution" }
          - else:
            output:    { power_state: "Nominal" }
        outputs:
          power_state: { type: string, enum: [Nominal, Caution, Critical, Emergency] }
        tests:
          - name: "nominal_steady_drain"
            input: { bms_frame: { version: 1, ts_ms: 1000, ... soc_pct: 95.0, cell_temp_c: 25.0 ... } }
            expect: { power_state: "Nominal" }
          - name: "thermal_caution"
            input: { ...cell_temp_c: 36.0 ... }
            expect: { power_state: "Caution" }
          # one entry per recorded NDJSON line
    END YAML

END ALGORITHM


ALGORITHM Trace_To_Test_Cases

    INPUT  ndjson_path: PathBuf
    OUTPUT yaml_test_cases: Vec<TestCase>

    SET frames ← read_lines(ndjson_path).map(parse_BmsFrame).collect()
    SET cases ← Vec::new()
    FOR EACH (idx, frame) IN frames.enumerate()
        SET expected ← classify_power_state(frame.cell_temp_c, frame.soc_pct)
        cases.push(TestCase {
            name: format!("recorded_frame_{:03}", idx),
            input: frame,
            expect: { power_state: expected },
        })
    ENDFOR
    RETURN cases

END ALGORITHM
```

Indent 4 spaces. The trace-to-test-cases pass is a one-shot Rust binary in `crates/stark-suit-station/src/bin/microgram-from-trace.rs` — reuses the lib's `BmsFrame` type and a copy of `classify_power_state` (or refactor to a `pub fn` in the lib if the import is cheap). Output is the YAML `tests:` block, written to stdout for human review before pasting into the microgram.

## Watch-outs

1. **Microgram schema is owned by rsk-core, not stark-suit** — read 3 existing micrograms before writing yours. Schema drift (calling a field `inputs` when rsk-core uses `params`, or `tests` when it uses `cases`) silently breaks the runtime without raising a parse error. This is the #1 way to lose a session.
2. **`classify_power_state` lives in `stark-suit-station/src/loops.rs` as a private fn** — promote it to `pub fn` in `stark_suit_station::bms` if the microgram-from-trace tool imports the lib. Otherwise duplicate the 4 lines verbatim with a comment pointing at the source. Pick one; do not let drift open up.
3. **NDJSON path indirection in YAML** — the microgram references the trace path. Use a workspace-relative path that resolves from the rsk-core root, not an absolute path. CI runners will mount the repo at a different prefix.
4. **The trace is not the contract; the rules are** — if `classify_power_state` thresholds change, the microgram's `tree:` must change in the same commit. Existing recorded traces stay valid only if their expected outcomes get recomputed. Bake a regen step into the microgram-from-trace binary.
5. **Don't widen rsk-core dependencies** — the microgram is a YAML file, not a Rust crate that imports stark-suit-station. The schema-link in `inputs.bms_frame.schema_ref` is documentary. If the microgram runtime needs to validate frame shape against the v1 schema, use a JSON Schema export of `BmsFrame` (cargo-supported via `schemars`) — do not make rsk-core build stark-suit-station.
